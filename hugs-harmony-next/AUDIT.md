# UX/UI Audit — ระบบการเงินคริสตจักรชีวิตสุขสันต์กาฬสินธุ์

> รอบที่ 1 ของกระบวนการ Redesign — เอกสารนี้เป็นฐานสำหรับรอบที่ 2 (ออกแบบใหม่ทั้งระบบ)
> วันที่ Audit: 16 ก.ค. 2026 · ขอบเขต: ทุกหน้าจอ + สถาปัตยกรรม + Design System

---

## 1. สรุปภาพรวม (Executive Summary)

ระบบปัจจุบันเป็น SPA แบบแท็บเดียว (8 แท็บ) ครอบคลุม workflow หลักของเหรัญญิกคริสตจักรได้ครบ
แต่มีปัญหา 3 ระดับ:

| ระดับ | ความรุนแรง | สรุป |
|---|---|---|
| **ความถูกต้องของข้อมูลการเงิน (Integrity)** | 🔴 วิกฤต | มี workflow ที่ "หลอก" ผู้ใช้ (อนุมัติแล้วไม่บันทึกจริง), ยอดนับจริงถูกทิ้ง, ประวัติรายละเอียดถูกลบเมื่อปิดยอด |
| **UX / Workflow** | 🟠 สูง | ไม่มี URL ต่อหน้า, ตารางใช้บนมือถือไม่ได้, ไม่มี confirm ก่อนการกระทำที่ย้อนกลับไม่ได้, งานซ้ำซ้อน 2 ที่ |
| **Design System / UI** | 🟡 ปานกลาง | สองระบบสไตล์ปนกัน, ตัวอักษรเล็กเกินไปสำหรับผู้ใช้สูงอายุ, สี hard-code หลุด token, ไม่มี dark mode |

## 2. ปัญหาวิกฤตระดับระบบ (ต้องแก้ก่อนเรื่องความสวยงาม)

### 2.1 แท็บ "แนบใบเสร็จ / ตรวจสอบ" เป็นระบบปลอม 🔴
- คิวรอตรวจสอบเก็บใน `localStorage` เท่านั้น (`hugs_pending_queue:v1`) — เห็นได้เฉพาะเครื่องเดียว ทั้งที่ workflow นี้มีไว้ให้ "คนหนึ่งส่ง อีกคนอนุมัติ"
- ปุ่ม **"อนุมัติ"** แค่ลบรายการออกจากคิว แล้วขึ้น toast ว่า *"อนุมัติและบันทึกบัญชีแล้ว"* — **ไม่มีการบันทึกลงบัญชีจริงแม้แต่บาทเดียว**
- รูปใบเสร็จเก็บเป็น base64 ใน localStorage → เต็ม quota เร็วมาก
- ตาราง `receipts` ใน DB มีอยู่แล้วแต่ไม่ถูกใช้เลย

### 2.2 ปิดยอดวันอาทิตย์ — ยอดนับจริงถูกทิ้ง 🔴
- ผู้ใช้นับธนบัตร/เหรียญจริง (`actualTotal`) แต่ตอนบันทึก `executeRollover` ส่ง `state.cash` (ยอดคำนวณ) ไปเป็น `close_cash` — **ยอดที่นับจริงไม่ถูกบันทึกที่ไหนเลย** ส่วนต่างเหลือแค่ข้อความอิสระ
- ปิดยอดแล้ว **ลบ** donation_rows และ expense_logs ของสัปดาห์ทิ้ง → เหลือแค่ยอดรวมใน history — ตรวจสอบย้อนหลัง (audit trail) ไม่ได้ ขัดกับหลัก stewardship/transparency
- ไม่มีหน้าสรุป + ยืนยันก่อนกดปิดยอด ทั้งที่เป็น action ที่ย้อนกลับไม่ได้
- ไม่มีการกระทบยอดฝั่งธนาคารเลย (นับแต่เงินสด)
- `date_closed` เก็บเป็น TEXT ภาษาไทย → เรียงลำดับ/ค้นหาไม่ได้

### 2.3 ความปลอดภัย 🔴
- Auth cookie เป็นค่า `"1"` ธรรมดา — ปลอมได้ทันที; รหัสผ่านเก็บ plaintext ใน env var
- RLS เปิด `anon_all` ทุกตาราง → ใครมี anon key อ่าน/เขียน/ลบข้อมูลการเงินได้หมดโดยไม่ต้อง login
- ไม่มีบทบาทผู้ใช้ (ศิษยาภิบาล / เหรัญญิก / กรรมการ ใช้สิทธิ์เดียวกันหมด) ทั้งที่ product นี้มีหลายบทบาทชัดเจน
- ปุ่ม "ลืมรหัสผ่าน?" และ "จดจำฉัน" เป็น UI ตาย ไม่ทำงานจริง

### 2.4 ความเสี่ยงข้อมูลสูญหาย 🟠
- `saveDonationRows` = ลบทั้งหมดแล้ว insert ใหม่ โดยไม่มี transaction — ถ้า insert ล้ม ข้อมูลทั้งสัปดาห์หาย
- เงินเก็บเป็น `REAL` (floating point) → คลาดเคลื่อนสตางค์สะสม ควรเป็น `NUMERIC` หรือสตางค์เป็น integer
- ตารางรายรับไม่มี autosave — กรอกทั้งเช้าวันอาทิตย์แล้วลืมกด "บันทึก" = หายหมด; ปุ่ม "เคลียร์" ไม่มี confirm
- ลบรายการเบิกจ่ายด้วย index + ไม่มี confirm dialog
- Webhook ยิงจาก client, ล้มแบบเงียบ, ไม่มี log/retry — automation มองไม่เห็นสถานะเลย

## 3. ปัญหาสถาปัตยกรรม Navigation

- ทั้งแอปเป็นแท็บใน `useState` — **ไม่มี URL ต่อหน้า**: refresh แล้วเด้งกลับแดชบอร์ด, ปุ่ม back ใช้ไม่ได้, แชร์ลิงก์หน้าเฉพาะไม่ได้
- Title ของ topbar กับ label ใน sidebar ใช้คนละข้อความ (เช่น "บันทึกใบเบิกจ่าย" vs "บันทึกรายจ่ายและการเบิกเงิน") — ผู้ใช้สับสนว่าอยู่หน้าไหน
- ลำดับเมนูไม่ตรง workflow จริงวันอาทิตย์ (รายรับ → เบิกจ่าย → ตรวจ → ใบอนุโมทนา → ปิดยอด) และมีปุ่ม logout ซ้ำ 2 ที่
- งานเบิกจ่ายกระจายอยู่ 2 แท็บที่ไม่เชื่อมกัน ("บันทึกใบเบิกจ่าย" เขียนลง DB, "แนบใบเสร็จ" อยู่ใน localStorage) — ผู้ใช้ไม่รู้ว่าต้องใช้อันไหน

## 4. Audit รายหน้าจอ

### 4.1 แดชบอร์ด
| ปัญหา | รายละเอียด |
|---|---|
| AI-pattern | Hero balance card + แถว stat cards 3 ใบ + progress bars = แพทเทิร์น dashboard สำเร็จรูปตามที่ brief ห้าม |
| ข้อมูลกำกวม | "สรุปกองทุน (สะสม)" รวม "สัปดาห์นี้ + ประวัติ" ปนกันโดยไม่บอกผู้ใช้ |
| ผิดที่ | "บันทึกประจำสัปดาห์" อยู่บนแดชบอร์ดแต่ถูกใช้ตอนปิดยอด — ควรอยู่ใน flow ปิดยอด |
| ตารางประวัติ | ใช้ div ไม่ใช่ `<table>`, ไม่มี sticky header, กดดูรายละเอียดรอบที่ปิดแล้วไม่ได้ (เพราะข้อมูลถูกลบ), ไม่มี pagination |
| สี | `FUND_BAR_COLORS` hard-code hex 7 สี หลุดออกนอก design tokens ทั้งหมด |

### 4.2 บันทึกรายรับ (Income)
| ปัญหา | รายละเอียด |
|---|---|
| มือถือใช้ไม่ได้ | ตารางบังคับ `min-w-[900px]` ทั้งที่การนับเงินวันอาทิตย์มักทำบนมือถือ/แท็บเล็ต — brief สั่งชัดว่าห้ามย่อ desktop |
| เสี่ยงข้อมูลหาย | ไม่มี autosave, ไม่มีตัวบอกว่า "มีข้อมูลยังไม่บันทึก", state ท้องถิ่นแยกจาก context (refresh ที่อื่นแล้ว desync) |
| พิมพ์ซ้ำ | ไม่มี autocomplete ชื่อสมาชิก — พิมพ์ชื่อเดิมใหม่ทุกสัปดาห์ ขัดหลัก "Minimal typing" |
| คอลัมน์ล้น | 7 กองทุนแสดงตลอด ทั้งที่แต่ละรายการมักกรอกแค่ 1–2 ช่อง |
| A11y | input ในตารางไม่มี label, ตัวเลือกช่องทางเป็น select จิ๋ว font 10px |

### 4.3 บันทึกใบเบิกจ่าย (Expense)
- ลบรายการโดยไม่มี confirm dialog
- ไม่แนบใบเสร็จได้ ทั้งที่มีแท็บแนบใบเสร็จแยกอยู่ (flow ขาดเป็น 2 ท่อน)
- Placeholder เป็นชื่อบุคคลจริง ("สุภาภรณ์ จิณวงษ์") — ไม่เหมาะสม
- ไม่มีหมวดหมู่รายจ่าย → รายงานแยกประเภทค่าใช้จ่ายไม่ได้เลย

### 4.4 แนบใบเสร็จ / ตรวจสอบ (Verification)
- ปัญหาวิกฤตตามข้อ 2.1 — ทั้งแท็บต้องออกแบบใหม่บนฐานข้อมูลจริง
- ฟอร์มซ้ำกับแท็บ Expense เกือบ 100% (ผู้เบิก/ผู้รับ/วัตถุประสงค์/จำนวน/ช่องทาง)
- input แก้ไขในการ์ดคิวไม่มี visual affordance ว่าแก้ได้
- ใช้ `any[]` — ไม่มี type safety

### 4.5 ใบอนุโมทนา (Receipts)
- พิมพ์ผ่าน `window.open` + `document.write` → โดน popup blocker, ไม่มี PDF
- ออกใบได้เฉพาะผู้ถวาย "สัปดาห์ปัจจุบัน" — ปิดยอดแล้วออกย้อนหลังไม่ได้ (ข้อมูลถูกลบ)
- ไม่มีเลขที่ใบ (running number), ไม่มีประวัติว่าออกไปแล้วบ้าง ทั้งที่อ้าง "ใช้ลดหย่อนภาษี" — ทางบัญชีถือว่าไม่สมบูรณ์
- ตาราง `receipts` ใน DB ไม่เคยถูกเขียน

### 4.6 ปิดยอดวันอาทิตย์ (Rollover)
- ปัญหาวิกฤตตามข้อ 2.2
- เครื่องนับธนบัตรเป็นไอเดียที่ดี ✅ ควรเก็บไว้และขยาย (บันทึก breakdown ธนบัตรลงประวัติ)
- ไม่มี step-by-step ทั้งที่เป็นงานที่เครียดที่สุดของสัปดาห์: ควรเป็น (1) ตรวจรายการ → (2) นับเงินสด → (3) กระทบยอดธนาคาร → (4) สรุป+ยืนยัน
- label ปุ่มเปลี่ยนความหมายตามสถานะ (สับสน) และไม่มี dialog ยืนยันขั้นสุดท้าย

### 4.7 ตั้งค่า (Settings)
- Webhook URL ปนกับ "ยอดยกมา" ในการ์ดเดียว — คนละเรื่องกันสิ้นเชิง
- แก้ยอดยกมาได้ตลอดเวลาแบบเงียบ ๆ → ยอดคงเหลือเพี้ยนโดยไม่มีร่องรอย ควรเป็น "รายการปรับปรุงยอด" ที่มีบันทึก
- "ล้างข้อมูลทั้งหมด" ทำลายทุกอย่างถาวรด้วย confirm คลิกเดียว — ควรบังคับพิมพ์ยืนยัน
- ไม่มี Automation Center ตาม brief (log, retry, API keys, หลาย integration) — มีแค่ URL เดียว

### 4.8 Login
- โครงสร้าง 2 panel + ข้อพระคัมภีร์ ทำได้ดี ✅
- "ลืมรหัสผ่าน?" / "จดจำฉัน" ไม่ทำงานจริง — dead UI ต้องตัดหรือทำให้จริง
- ไม่มี CSRF/session จริง (ดูข้อ 2.3)

### 4.9 คู่มือ (Help)
- FAQ คงที่ 5 ข้อ พออยู่ได้ แต่ควรเป็นคู่มือตาม workflow (step-by-step ประจำวันอาทิตย์) มากกว่า Q&A

## 5. Audit Design System

| หัวข้อ | สถานะ | ปัญหา |
|---|---|---|
| Tokens | 🟡 | `tokens.css` (Hallmark) ดี แต่มี alias layer `--c-*` ซ้อน + สี hard-code ใน component หลุดระบบ |
| สองระบบสไตล์ | 🔴 | Dashboard/Login/Shell ใช้ semantic CSS classes; แท็บอื่นใช้ Tailwind utilities — ปนกันไม่มีกติกา |
| Typography | 🔴 | meta text 10–11px เกลื่อนระบบ ทั้งที่ผู้ใช้จำนวนมากเป็นอาสาสมัคร/ผู้สูงอายุ — ควร base 14–16px |
| Touch targets | 🔴 | ปุ่มลบ `p-1` (~22px), select จิ๋ว — ต่ำกว่ามาตรฐาน 36–44px ที่ design.md กำหนดเอง |
| Icons | 🟡 | inline SVG path ซ้ำ (ไอคอนถังขยะประกาศซ้ำ 3+ ที่) — ไม่มี icon component กลาง |
| Dark mode | ❌ | ไม่มี ทั้งที่ brief กำหนด |
| Tables | 🔴 | ไม่มี sticky header, sort, filter, keyboard nav, bulk actions — ห่างจาก "world-class table" ใน brief มาก |
| Dialog A11y | 🟠 | ConfirmDialog ไม่มี focus trap, ไม่ปิดด้วย Esc, ไม่มี `role="dialog"` |
| Motion | 🟡 | มี fade-in พื้นฐาน; ไม่มี reduced-motion ครบตาม design.md |

## 6. สิ่งที่ทำได้ดี — ควรเก็บไว้

1. **โทน warm editorial (paper/ink/amber)** — เหมาะกับความรู้สึก "อบอุ่น น่าเชื่อถือ" ของคริสตจักร ควรต่อยอด ไม่ทิ้ง
2. **เครื่องนับธนบัตร/เหรียญ** ในหน้าปิดยอด — เข้าใจ workflow จริง
3. **แนวคิดกระทบยอด (expected vs actual + ผลต่าง)** — ถูกหลักการ แค่บันทึกผลไม่ครบ
4. **ใบอนุโมทนามีคำอวยพร + อ้างอิงพระคัมภีร์ที่ Login** — ภาษาแบบคริสตจักร ไม่ corporate
5. Sarabun + Inter คู่ฟอนต์ไทย/ละตินเหมาะสมแล้ว

## 7. พิมพ์เขียวสำหรับรอบที่ 2 (Redesign Blueprint)

### 7.1 Information Architecture ใหม่ (เรียงตาม workflow จริง)
```
/                    ภาพรวม (ยอดคงเหลือ + สิ่งที่ต้องทำสัปดาห์นี้ + กิจกรรมล่าสุด)
/income              รายรับสัปดาห์นี้ (autosave, mobile-first, autocomplete ชื่อ)
/expenses            รายจ่าย + แนบใบเสร็จ + คิวอนุมัติ (รวมเป็น flow เดียว, สถานะ: ร่าง→รออนุมัติ→อนุมัติ)
/receipts            ใบอนุโมทนา (มีเลขที่, ประวัติการออก, ออกย้อนหลังได้)
/closing             ปิดยอดวันอาทิตย์ (wizard 4 ขั้น, เก็บ breakdown, ไม่ลบข้อมูลดิบ)
/history/[week]      เจาะดูรายละเอียดรอบที่ปิดแล้ว (audit trail สมบูรณ์)
/automation          Automation Center (integrations, webhook logs, retry, API keys)
/settings            ข้อมูลคริสตจักร, กองทุน, ผู้ใช้/บทบาท, การปรับปรุงยอด (มีบันทึก)
```

### 7.2 หลักการแก้ระดับข้อมูล
- ปิดยอดแล้ว **ไม่ลบ** รายการดิบ — ใช้สถานะ `closed` + สัปดาห์ผูกกับรายการ
- บันทึกยอดนับจริง + breakdown ธนบัตรลงประวัติ
- เงินเป็น NUMERIC, วันที่เป็น timestamptz
- คิวอนุมัติอยู่ใน DB + อนุมัติแล้วสร้างรายการจ่ายจริง
- Auth จริง (role: admin / treasurer / viewer) + ตัด `anon_all`

### 7.3 หลักการแก้ระดับ UI
- ระบบสไตล์เดียว (Tailwind + tokens), ตัด alias layer
- Base font ≥ 14px, ตัวเลขเงิน tabular-nums ขนาดใหญ่, touch target ≥ 44px
- ตารางการเงิน: sticky header, sort, ค้นหา, export, แถวรวมชัดเจน
- มือถือ: bottom navigation + card-based entry แทนตาราง 900px
- Confirm ทุก action ทำลายข้อมูล; typed-confirm สำหรับล้างข้อมูล
- Dark mode ผ่าน tokens
